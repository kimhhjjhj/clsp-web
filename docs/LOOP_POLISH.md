# CLSP 완성도 루프 (Polish Loop)

> 새 기능 추가가 아닌, **이미 있는 화면을 프리미엄급으로 다듬는** 자동 루프.
> `LOOP_PROMPT.md`(기능 백로그)와 **병행** 사용. 서로 간섭하지 않음.
>
> **목표**: "수억원 들인 상용 제품"처럼 보이고 느껴지는 완성도.

---

## 실행 방법

### 수동
사용자가 "다음 폴리시", "계속", "ㄱㄱ" 이라고 지시할 때마다 아래 본문 1회 실행.

### `/loop` + 이 프롬프트 본문
`/loop` 슬래시 커맨드의 인자로 아래 "루프 본문"을 그대로 붙여넣기.

### 중단 조건
- 사용자가 새 방향 지시
- 연속 3사이클 "개선 체감 없음" — Polish 루프 자율 종료
- 기능상 블로커 (LOOP_PROMPT로 인계)

## 루프 본문 (복붙용)

```
CLSP 완성도 다듬기 사이클을 1회 실행한다. 새 기능 추가 금지.

## 스텝

### 1. 대상 선정
이미 선택된 페이지가 없다면 아래 **대상 로테이션**에서 하나 선택.
같은 페이지를 3사이클 이상 연속 선택하지 말 것 (다른 영역으로 강제 전환).

**대상 로테이션 (순환)**
1. 대시보드 (`/`)
2. 프로젝트 목록 (`/projects`)
3. 프로젝트 상세 (`/projects/[id]`)
4. Stage 2 · 프리콘 (`/projects/[id]/stage/2`)
5. Stage 3 · 시공관리 (`/projects/[id]/stage/3`)
6. Stage 4 · 분석준공 (`/projects/[id]/stage/4`)
7. 사업 초기 검토 (`/bid`)
8. 전사 분석 (`/analytics`)
9. 생산성 DB (`/standards`)
10. R&O 라이브러리 (`/risks`)
11. 협력사 (`/companies`)
12. 엑셀 임포트 (`/import`)
13. 관리자 승인 (`/admin/productivity`)
14. 프로젝트 생성 (`/projects/new`)
15. 프로젝트 수정 (`/projects/[id]/edit`)
16. 일보 작성 (`/projects/[id]/daily-reports/new`)
17. 일보 상세 (`/projects/[id]/daily-reports/[did]`)
18. 사이드바 · 상단바 (모든 페이지 공통)
19. 모바일 뷰 (viewport 375px 가정, 위 대상 중 1~5 재검사)

### 2. 진단
선택한 대상에 대해 아래 체크리스트로 평가 (0~5점, 1분 이내):

- [ ] **빈 상태 품격**: 0건·1건·많음 모두 자연스러운가
- [ ] **로딩 경험**: skeleton이 실제 레이아웃을 예고하는가 (덩그러니 스피너 X)
- [ ] **시각 위계**: 가장 중요한 숫자·상태가 0.5초 안에 읽히는가
- [ ] **색 일관성**: 다른 페이지의 같은 역할 요소와 색·크기 일치하는가
- [ ] **여백 리듬**: 섹션 간격·내부 패딩이 균일한가 (들쭉날쭉 X)
- [ ] **상호작용**: 호버·포커스·트랜지션이 있는가 (딱딱한 상태변화 X)
- [ ] **반응형**: sm/md/lg/xl에서 깨지지 않는가 (특히 모바일 너비 375px)
- [ ] **에러 상태**: API 실패·권한 거부·네트워크 끊김에 메시지 있나
- [ ] **CTA 가시성**: 이 페이지의 "다음 액션"이 명확히 보이는가
- [ ] **터치 타깃**: 모바일 기준 최소 40×40px 확보했는가

2점 이하 항목이 1개라도 있으면 그게 이번 사이클의 작업 대상.
모두 4점 이상이면 "이 페이지 OK" 선언하고 다음 대상으로 넘긴다.

### 3. 계획
선택한 2점 이하 항목 1~2개에 대해:
- 어떤 파일·컴포넌트 수정할지 식별
- 공통 컴포넌트(`components/common/*`)에 들어갈 만한 건 그쪽으로 추출
- 새 기능 요구가 떠오르면 `LOOP_PROMPT` 백로그에 추가하고 이 사이클은 건드리지 않음

### 4. 구현
- **한 사이클 ≤ 3파일** 원칙 (너무 많이 한번에 바꾸지 말 것)
- 기존 디자인 토큰(StatusBadge, PageHeader accent, card-elevated 등) **먼저 재사용**
- 모호한 디자인 판단은 사용자 확인 없이 진행 (나중에 되돌릴 수 있으니)
- 색·크기 임의 값보다 Tailwind 표준·CSS 변수 우선

### 5. 검증
- `npx tsc --noEmit` 으로 타입 오류 없는지 (관련 파일 grep)
- dev 서버가 있으면 해당 페이지 `curl -o /dev/null -w "%{http_code}"` 로 200 확인
- 빈 상태/데이터 있음 두 케이스 모두 확인 (있다면)
- 모바일 시뮬: User-Agent `iPhone` 또는 devtools 375px에서 시각 회귀 없음

### 6. 커밋·푸시
- 커밋 메시지:
  · `style:` 또는 `polish:` 접두사 (새 기능 아님을 명시)
  · 제목 1줄: 대상 페이지 + 개선 요점
  · 본문: Before/After 한 줄씩, 수정 파일 1~3개
  · 마지막: `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>`
- `git push` 까지 자동

### 7. 보고 (2~4줄)
1. 어떤 페이지의 어떤 항목을 다듬었는지
2. 눈에 띄게 바뀐 것 1~2개
3. 같은 페이지에 남은 2점 이하 항목 있으면 명시
4. 다음 사이클 예정 대상

## 원칙

- **Polish 루프는 새 기능 금지**. 기능 아이디어는 `LOOP_PROMPT` 백로그로 이관.
- 변경은 작게, 눈에 보이게. "그럴싸한 1%" 같은 무의미한 미세조정 금지.
- 체크리스트의 모든 항목이 4점 이상이면 그 페이지는 "OK" 선언하고 다음 대상.
- 같은 페이지 3사이클 연속 금지. 로테이션을 강제로 돌린다.
- 이미 공통 컴포넌트가 있는데 페이지 로컬 스타일을 새로 만들지 말 것.
  (예: StatusBadge 있는데 또 span 직접 만들지 마라)

## 완료 조건

- 대상 로테이션의 **모든 19개 대상**에서 체크리스트 대부분 항목이 4점 이상
- 이때 "Polish 1순회 완료" 라고 보고하고 루프 자발 종료
- 필요하면 2순회 시작 (더 엄격한 기준으로)
```

## 체크리스트 스코어 기준

각 항목은 0~5점:
- **5**: 상용 제품 수준 (Linear/Notion/Stripe 급)
- **4**: 프로 수준 (평범한 SaaS 이상)
- **3**: 괜찮음 (중간)
- **2**: 어색함 (개선 필요)
- **1**: 나쁨 (즉시 고쳐야)
- **0**: 아예 없음/깨짐

## 세션 간 상태 유지

매 사이클 끝에 `scripts/polish-state.json` 같은 파일에:
```json
{
  "lastTarget": "/projects/[id]/stage/3",
  "roundNumber": 1,
  "scoreHistory": [
    { "target": "/", "date": "2026-04-19", "scores": {...}, "status": "OK" }
  ]
}
```
기록하면 여러 세션에 걸쳐도 로테이션 유지 가능 (선택 구현).
